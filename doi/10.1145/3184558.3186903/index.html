POSTER: Predicting Website Abuse Using Update Histories

Yuta Takata NTT Secure Platform Laboratories
Mitsuaki Akiyama NTT Secure Platform Laboratories
Takeshi Yagi NTT Secure Platform Laboratories
Kunio Hato NTT Secure Platform Laboratories, takata.yuta@lab.ntt.co.jp
Shigeki Goto Waseda University, goto@goto.info.waseda.ac.jp

DOI: https://doi.org/10.1145/3184558.3186903
WWW '18: Proceedings of The Web Conference 2018, Lyon, France, April 2018

Threats of abusing websites that webmasters have stopped updating have increased. In this poster, we propose a method of predicting potentially abusable websites by retrospectively analyzing updates of software that composes websites. The method captures webmaster behaviors from archived snapshots of a website and analyzes the changes of web servers and web applications used in the past as update histories. A classifier that predicts website abuses is finally built by using update histories from snapshots of known malicious websites before the detections. Evaluation results showed that the classifier could predict various website abuses, such as drive-by downloads, phishes, and defacements, with accuracy: a 76% true positive rate and a 26% false positive rate.

1 Introduction

Threats of hacking other people's websites have been increasing. Websites are hacked by abusing vulnerabilities in web servers and web applications or stealing account credentials by using malware. Especially, abuses of the CMS (Content Management System) pervasively used by many webmasters are increasing year by year.

To prevent such website hacking, it is effective to have webmasters update the version of CMS and apply security patches. However, many webmasters do not update software or check security configurations for a long time, and these abandoned websites are abused by attackers [8]༎ Security warnings must be sent to the webmasters on the basis of the current website status such as software updates and security configurations before abuse by attackers [6].

In this poster, we focus on update histories of software that composes a website. We propose a method of predicting potentially abusable websites by retrospectively analyzing whether the webmasters have appropriately managed their own websites. This method accesses archived snapshots of past websites and analyzes the changes of web servers and web applications used so far as update histories. Finally, it builds a classifier that predicts website abuses by using update histories of known malicious websites before the detections. In our experiment, we used 397,629 snapshots of 10,730 URLs for the classifier. Evaluation results showed that the classifier could predict website abuses with accuracy: a 76% true positive rate (TPR) and a 26% false positive rate (FPR).

2 Proposed Method

2.1 Collecting Archived Snapshots

Internet Archive provides a service, called Wayback Machine [4], for archiving past websites (snapshots). Wayback Machine archives web content collected by crawling websites that are well linked to from other websites. Therefore, we can identify web servers and web applications used in the past if we can access snapshots archived by Wayback Machine.

2.2 Compiling Update Histories

We trace an update history by comparing features listed in Table 1 in a time series. If the type of extracted features is Strings, the update history has four categories: 1. “Up” means the value began to be observed; 2. “Stable” means the observed value did not change; 3. “Updated” means the observed value changed; and 4. “Down” means the value was no longer observed. If the type of extracted features is Boolean, the update history has three categories: 1. “Up” means the value changed from False to True; 2. “Stable” means the value did not change; and 3. “Down” means the value changed from True to False. Figure 1 shows two example update histories created from three archived snapshots. More precisely, Update History 0 is created by comparing Archived Snapshot 0 with Archived Snapshot 1, and Update History 1 is created by comparing Archived Snapshot 1 with Archived Snapshot 2. Then, a time interval Δ between archived snapshots was also included in these update histories by comparing the number of months in archived Timestamp.

2.3 Designing Feature Vectors

We design a feature vector used for our classifier by combining update histories in the previous subsection. In other words, the number of a feature vector's dimensions is determined by the number of update histories. Therefore, to align the number of dimensions, we predefined the number of times to compare archived snapshots N when compiling update histories. Note that we used a missing value (e.g., − 1) when we could not compare for the predefined number of times due to the number of archived snapshots.

3 Evaluation

3.1 Experimental Environment and Datasets

In this experiment, we collected datasets in three steps using the experimental environment shown in Fig. 2. First, our crawler accesses inputted seed URLs and identifies CMS types of the websites on the basis of the features listed in Table 1. In this experiment, we focus on the top three CMSs: WordPress, Drupal, and Joomla!. Next, we collected archive URLs of websites whose CMS types were identified using Time Travel API [3]. Finally, the crawler accesses the archive URLs and collects snapshots used for compiling update histories.

Using this environment, we collected 341,431 snapshots of URLs listed in Alexa Top 250K Sites [1] as a benign dataset. We also collected 56,198 snapshots of three types of malicious URLs as a malicious dataset: URLs from which our high-interaction honeyclients detected drive-by downloads, phishing URLs listed in PhishTank [2], and defacement URLs listed in Zone-H [5].

3.2 Evaluating Detection Capability

In this evaluation, we measured the accuracy of our classifier built by random forest using learning data and test data separated from the datasets in the previous subsection by 10-fold cross validation. Then, we defined the number of times to compare archived snapshots as n and the minimum time interval (the number of months) between archived snapshots as δ. Table 2 shows the calculated results of AUC (Area Under Curve) values with various parameters. As shown in Table 2, the larger the number of comparisons n, the longer the time interval δ, the higher the AUC value. This means that we can detect more website abuses on the basis of webmaster behaviors by retrospectively analyzing websites rather than analyzing websites at the certain time.

3.3 Evaluating Prediction Capability

In this subsection, we separated malicious datasets into learning data that was detected before a certain time and test data that was detected after the certain time and evaluated the accuracy the same way as in the previous subsection. This method can evaluate the capability for predicting whether websites in the test data are potentially abusable in the future at the certain time [7]. In this experiment, we separated datasets from January 2017. From the results of evaluating the accuracy of our classifier with the best performance parameters n = 6, δ = 8, the AUC was 0.803. Compared with the prediction capability of the existing method (AUC=around 0.7) [7], our proposed method achieved better accuracy.

4 Conclusions

We proposed a method of predicting website abuses by retrospectively analyzing update histories of software that composes a website by using Internet Archive. Evaluation results showed that a classifier built by the method could predict website abuses with accuracy: a 76% TPR and a 26% FPR. Using such accurate prediction results, we can prevent website abuses by preliminarily sending security warnings to webmasters.
